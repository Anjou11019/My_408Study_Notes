##### 堆、栈及内存管理

**栈（stack）**：存在于某作用域（scope）的一块内存空间（memory space）。当你调用函数幂函数本身即会形成一个stack用来放置他所接受的参数及返回地址以及local object

在函数本体内部（function body）声明的任何变量，其所使用的内存块都取自上述stack

**堆（heap）**：也叫system heap，有操作系统提供的一块公共的（global）内存空间，程序可动态分配（dynamic allocated）从某中获得若干区块（blocks）

```cpp
class Complex{....}
...
{
    Complex c1(1,2);
    Complex* p = new Complex(3);
    static Complex c2(1,2);

...
    delete p;
}
Complex c3(1,2)
```

上述例子，c1所占的空间就来自stack，离开作用域的时候c1自然消亡，Complex(3)则是以new的形式动态获得的，需要手动释放空间

c1便是所谓的**stack object**，其生命在作用域结束之后结束，这种作用域内的object，又叫做auto object，因为它会被自动清理

c2便是所谓**static object**，其生命在作用域结束之后仍然存在，直到整个程序结束

c3便是**global object**，其生命在整个程序结束之后才结束。可以视为一种 **static object**，其作用域是整个程序

p所指便是**heap object** ，其生命在它被delete之际才结束，若不delete， 则会内存泄漏（memory leak），当作用域结束，其所指的 **heap object** 仍然存在， 但指针p的生命结束了，作用域之外再也看不到**p**也就没机会delete

##### 不同问题及解法（23个设计模式之一）

**adapter（适配器）**：

问题背景 ： 
已有类接口与当前系统所需接口不兼容，但又不希望修改原有类（可能是第三方库、遗留代码）

核心思想  ：
在“客户端”和“被适配对象”之间增加一层转换层，将旧接口转换成新接口。属于结构型模式

**pImpl（point to implementation）**：

问题背景  ：
类的实现细节暴露在头文件中，导致：

    编译依赖严重（头文件改一点，全部重编译）

    无法隐藏私有实现

    二进制兼容性差

核心思想  ：
    将实现细节隐藏到一个内部实现类中，头文件只暴露指针。属于封装与编译隔离技巧。

**template method（模板方法）**：

问题背景  ：
    一个算法的整体流程固定，但某些步骤由子类决定。

核心思想 ： 
    在父类中定义算法骨架，某些步骤声明为 virtual，由子类实现
    属于行为型模式

**Oberver（观察者）**：

 问题背景 ： 
    一个对象状态变化，需要通知多个依赖对象，但又不希望强耦合。

核心思想  ：
    定义一对多依赖关系，状态改变时自动通知观察者。

**Composite（组合模式）** ：让“单个对象”和“对象集合”可以被统一对待

问题背景 
    希望“单个对象”和“对象集合”在使用时一致。

核心思想 
    定义统一的 Component 接口。

        Primitive（叶子节点）

        Composite（容器节点，内部维护子节点列表）

**Prototype（原型模式）**:

问题背景 
    对象创建成本高，或者创建过程复杂，或者不希望依赖具体类名。

核心思想 
    通过 clone() 复制已有原型对象，而不是 new 某个具体类。

关键机制

    父类定义 virtual clone()

    子类实现 clone()

    父类维护 prototype 注册表

    客户端通过 findAndClone(type) 获取对象

##### 重载new(),delete()  placement new

可以重载class member operator new(),写出多个版本，前提是每一个版本的声明都必须有**独特的参数列**，其中第一个参数必须是**size_t**,其余参数以new所指定的placement arguments为初值。出现于new(......)小括号内的便是所谓placement arguments

```cpp
Foo* pf = new(300,'c')Foo;//实际参数是3个
```

我们也可以重载class member operator delete(),写出多个版本。但他们绝不会被delete调用。**只有当new所调用的ctor抛出exception，才会调用这些重载版的operator delete().** 它只可能这样被调用，主要用来释放未能完全创建成功的object所占用的memory（简单说：在placement new的时候系统先分配内存再调用构造函数，但是在构造函数抛出异常的时候需要有一个对应的placement delete去释放掉这些内存）

即使operator delete未能一一对应operator new，也不会有任何报错（你的意思是：放弃处理ctor发出的异常）
